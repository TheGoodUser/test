Why do you prefer Riverpod over Provider or Bloc?

▼

Riverpod gives me compile-time safety and better testability. With Provider, you can have runtime errors if you forget to wrap a widget with the provider. Riverpod doesn't need a BuildContext for most things, which makes the code cleaner.

I haven't used Bloc extensively, but from what I understand, it's great for complex state machines. For most apps, though, Riverpod's approach feels more straightforward without too much boilerplate.

How do you handle API calls in Flutter?

▼

I typically use the http or dio package depending on the requirements. For simple REST APIs, http works fine. For anything that needs interceptors, retries, or file uploads, I go with dio.

I usually create a service layer that returns typed models, handle errors with custom exceptions, and use FutureBuilder or AsyncValue (if using Riverpod) to manage loading states in the UI. For real-time data, I've used WebSockets and Supabase's realtime subscriptions.

Explain the widget lifecycle in Flutter

▼

For StatelessWidget, there's just the build method—pretty straightforward.

For StatefulWidget, it's more involved:

• createState() creates the state object
• initState() runs once when the widget is inserted into the tree—this is where I set up controllers, fetch initial data
• didChangeDependencies() runs after initState and whenever the dependencies change
• build() gets called whenever setState is triggered or the parent rebuilds
• dispose() is where I clean up controllers, listeners, subscriptions to avoid memory leaks

I've debugged enough memory issues to be religious about disposing controllers.

What's the difference between mainAxisAlignment and crossAxisAlignment?

▼

In a Column, mainAxis is vertical (top to bottom), crossAxis is horizontal. In a Row, it's flipped—mainAxis is horizontal, crossAxis is vertical.

MainAxisAlignment controls spacing along the main axis (like spaceBetween, center, start). CrossAxisAlignment controls alignment perpendicular to that (stretch, center, start).

How would you optimize a Flutter app that's lagging?

▼

First, I'd profile it using Flutter DevTools to see where the bottleneck is. Common fixes:

• Use ListView.builder instead of ListView with a list of children—lazy loading matters
• Avoid rebuilding the entire widget tree; use const constructors where possible
• If I'm using complex animations, I'd use RepaintBoundary to isolate expensive widgets
• For images, use cached_network_image and proper sizing
• Check if I'm doing heavy computations in the build method—move those to isolates if needed
• Sometimes it's about reducing unnecessary setState calls or splitting widgets better

What's the difference between async and async*?

▼

async returns a Future—you use it for one-time asynchronous operations, like an API call.

async* returns a Stream—it can yield multiple values over time. For example, if I'm listening to Firestore updates or tracking user location changes, I'd use async* with yield to emit values as they come in.

You've worked with FastAPI. Why FastAPI over Flask or Django?

▼

Speed and type safety. FastAPI uses Python type hints for automatic validation and documentation with OpenAPI. It's built on Starlette and Pydantic, which makes it async-first—important when you're handling concurrent requests.

Django is great for monolithic apps with admin panels, but for pure APIs, FastAPI feels leaner. Flask is fine too, but you end up writing a lot of boilerplate that FastAPI handles out of the box.

Explain how you implemented JWT authentication in your intrusion detection project

▼

I used HTTP-only cookies to store JWT tokens—this prevents XSS attacks since JavaScript can't access the cookie. On login, the server generates a JWT with user claims and sends it as a cookie with httpOnly=True, secure=True, and sameSite='Lax'.

For protected routes, I have a dependency in FastAPI that reads the cookie, verifies the token signature using the secret key, and extracts the user info. If the token's expired or invalid, it returns a 401. For refresh tokens, I issue a longer-lived token that can generate new access tokens.

What's the difference between authentication and authorization?

▼

Authentication is verifying *who* you are—like logging in with a password or OTP.

Authorization is checking *what* you're allowed to do after you're authenticated. For example, in the HRMS app, authentication happens when an employee logs in. Authorization determines whether they can approve leave requests (manager role) or just view their own salary slips (employee role).

How did you integrate Razorpay in your project?

▼

It's a two-step process. First, the backend creates a Razorpay order using their API—you send the amount, currency, and get back an order ID. This order ID is sent to the Flutter app, which opens the Razorpay checkout.

After the user completes payment, Razorpay sends a signature back to the app. The app forwards this to the backend, which verifies the signature using HMAC-SHA256 with your API secret. If it matches, the payment is legit, and you update the database. I never trust the frontend to confirm payment status—verification always happens server-side.

What security measures did you implement in your projects?

▼

Several layers:

• Input validation on both frontend and backend—never trust client data
• Parameterized queries to prevent SQL injection
• JWT tokens in HTTP-only cookies to prevent XSS
• CORS properly configured to allow only trusted origins
• Rate limiting to prevent brute-force attacks—I track failed login attempts by IP
• Row-level security in Supabase to isolate tenant data
• Environment variables for secrets, never hardcoded
• HTTPS everywhere for data in transit

What's row-level security in Supabase?

▼

RLS lets you define policies directly in the database that restrict which rows a user can access. For example, in a multi-tenant app, you'd have a policy like "users can only select rows where user_id = auth.uid()".

Even if someone bypasses your application logic or uses the API directly, the database enforces these rules. It's defense in depth—your last line of security.

How do you debug a production crash?

▼

First, I check crash logs—Firebase Crashlytics or Sentry if it's integrated. The stack trace usually points to the exact line. If it's a null pointer or a state issue, I try to reproduce it locally.

I add more logging around the suspected area, sometimes add try-catch blocks temporarily to isolate the problem. If it's a rare race condition, I'd look at threading issues or async state changes. Once I find the root cause, I fix it, write a test to prevent regression, and push a hotfix.

Explain a challenging bug you fixed

▼

In the HRMS app, we had this weird issue where the attendance screen would sometimes show duplicate entries. Turned out, the backend was sending paginated data, and our Flutter app was appending new data to the list on scroll.

But if the user scrolled up and triggered a refresh, we weren't clearing the old list—just kept appending. The fix was to differentiate between "load more" (append) and "refresh" (replace). Added a flag to track pagination state and cleared the list on pull-to-refresh. Simple in hindsight, but debugging took a while because it only happened with specific scroll patterns.

Interview tips

  • Don't oversell: If you haven't done something, say so. "I haven't used Bloc extensively, but I understand the pattern and can pick it up quickly."
  • Be specific: Instead of "I optimized the app," say "I replaced ListView with ListView.builder and added caching, which cut load time from 2.1s to 1.2s."
  • Ask questions: At the end, ask about their tech stack, team structure, or types of projects you'd work on. Shows genuine interest.
  • Prepare your workspace: Since it's remote, test your mic/camera beforehand. Have your resume and a notepad handy.
  • Code on the spot: They might ask you to share your screen and write Flutter code. Practice building a simple list view or API call live.
